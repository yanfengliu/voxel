# 0.2 foundation migration note

Status: unreleased work on the path from 0.1.4 to 0.2. Update this note whenever the public API report changes during the milestone.

## Existing consumers

Existing snapshot-only consumers that use Three.js `WebGLRenderer` ordinarily require no source change:

- The `voxel.render-snapshot/1` literal and every existing required snapshot/resource/chunk/batch field are unchanged. `WorldDescriptorV1` adds optional `chunkProfile` and `transactionLimits` fields, and `InstanceBatchV1` adds an optional neutral `presentation` shadow-participation policy.
- `RenderWorld.acceptSnapshot`, accepted/pending/presented accessors, revision properties, `markPresented`, and `dispose` remain available. Validation is stricter: within one world/epoch, an existing `{key, incarnation, revision}` tuple must keep identical content, item revisions may not regress, and a key recreated after removal must exceed its lane tombstone. Snapshot-only callers need no source edit, but stale ABA reuse or changed content at the same item revision now rejects instead of being accepted ambiguously.
- Existing `ThreeRenderRuntime` construction call shapes and the `acceptSnapshot`, `frame`, `resize`, `setView`, `capture`, `metrics`, and `dispose` call patterns remain source-compatible. `acceptSnapshot` now accepts `unknown` so runtime validation owns narrowing, and a successful `frame` may return an immutable presented manifest; existing callers that pass snapshots and ignore the frame result do not need to change.
- Public RenderWorld snapshot accessors still return caller-owned defensive copies.

Custom `RendererLike` adapters and test doubles must add `getSize(target: Vector2)` and `getPixelRatio()`. This pre-1.0 contract change lets construction and resize restore a borrowed runtime-owned viewport exactly after failure. Three.js `WebGLRenderer` already implements both methods.

The implementation now prepares one canonical owned snapshot for Three validation and presentation. This removes repeated retained-array copying without allowing a caller to mutate canonical state.

`ThreeRenderRuntime.metrics()` now includes cumulative snapshot input/copied bytes and copy operations, defensive snapshot-copy bytes, current/peak retained canonical bytes, and current/peak presentation-staging bytes. Ordinary canonical presentation lanes add no staging allocation. Profiled mesh outputs count their unique backing-buffer capacity while provisional, accepted-but-pending, or retained by a frame ticket; committed buffers are excluded. Peak staging records replacement and rejected-candidate overlap. A committed buffer leaves current staging while any newer pending or ticket-retained output remains counted; abort and context loss retain pending CPU staging, and disposal clears the current gauge. Rejected backend candidates contribute to input/copy work but never to canonical retained bytes.

Tests that need deterministic core counters can import `readRenderWorldOwnershipMetricsForTesting` and `resetRenderWorldOwnershipMetricsForTesting` from `voxel/testing`. Reset preserves the current retained-byte value and makes it the new peak baseline; it never changes world state.

## Capability discovery

Consumers that need to branch on implemented backend support may use either:

    import { getThreeRuntimeCapabilitiesV1 } from 'voxel/three';

    const capabilities = getThreeRuntimeCapabilitiesV1();

or an existing runtime instance:

    const capabilities = runtime.getCapabilities();

The returned object is immutable and process-wide. It reports snapshot and Delta V1 ingest, triangle geometry presentation, opaque visible-face voxel chunks, isometric orthographic, perspective, and borrowed-camera policies, plus runtime-rendered and embedded host modes with explicit draw/capture/viewport ownership. It still reports no committed runtime picking, no asynchronous worker presentation, and event/resize context handling rather than complete GPU reconstruction; packaged worker and scheduler primitives alone do not change those runtime claims.

Do not infer future roadmap features from portable core types. In particular, portable validation of line or point topology does not mean the current Three instance backend supports it; consult the capability report and handle atomic rejection.

## Browser worker startup

Browser bundlers should import `startBrowserMeshWorkerV1` from `voxel/meshing/browser-worker`. That dedicated DOM entry uses the static `new Worker(new URL(...))` form needed for bundlers to emit the packaged worker asset. The primary `voxel/meshing` entry remains DOM-free; portable or custom hosts continue to use `startMeshWorkerV1(factory)` there.

Synchronous worker construction returns a `started` result with the owned worker handle or a bounded `worker-startup-failed` diagnostic. Module fetch or evaluation can still fail later through the worker's `error` event; the scheduler/runtime owner must route that event into its failure policy and terminate every started handle on disposal.

## Atomic scheduler target admission

`VoxelMeshSchedulerV1.enqueueTarget(groups, logicalTick)` is an additive alternative to `enqueue` for callers that already know every dependency group required by one world, epoch, and revision. It validates the complete group set, combined queue cost, and simultaneous staging-lease budget before registering or superseding any work. Existing callers may continue using `enqueue` for one scheduler group and require no source change; revision-atomic runtime integration uses the target form so a rejected later group cannot leave a partially admitted target.

Use `enqueueReplacingEpochTarget(groups, logicalTick)` when that complete target also establishes a new authoritative world epoch. It performs target validation and budget preflight before cancelling prior-epoch groups or refreshing worker generations, so rejection leaves the old epoch and its work unchanged. Existing `replaceEpoch` plus `enqueueTarget` callers remain supported, but that two-call sequence intentionally exposes an intermediate replaced state and is not suitable for an atomic presentation coordinator.

## Public API review

Built public declarations are recorded in api/public-api.json. An intentional declaration change requires all of:

1. Review the emitted declaration diff and compatibility impact.
2. Update this note or the relevant migration document.
3. Add a changelog entry.
4. Run npm run api:update.
5. Run npm run test:api and the packed-consumer checks.

The API report is a change detector, not proof that a changed type is compatible. Review and consumer compile fixtures remain required.

The 0.2 package emits ESM and declarations without source-map directives and excludes `.map` files from the tarball. The packed-consumer gate rejects both map files and dangling directives, and enforces the provisional 350,000-byte packed and 1,700,000-byte unpacked budgets. Disabling declaration maps changes the committed declaration hashes but not their exported TypeScript semantics.

## Remaining 0.2 additions

The packed single-Three consumer, ownership metrics, and exact TypeScript 5.7.3/5.9.3/6.0.3 compatibility fixtures are implemented. The City-shaped fixture now executes an initial building snapshot plus sparse generational batch delta through a borrowed renderer/scene, perspective camera, and host-owned prepare/draw/commit/capture/resize loop. It verifies neutral shadow participation, geometry bounds, sparse upload work, accepted/presented ordering, and teardown ownership. `frame()` remains the standalone compatibility path; embedded hosts use single-use `prepareFrame` tickets followed by exactly one host draw and `commitFrame`, or `abortFrame` when no draw occurs.

A first remote CI run and final foundation evidence remain before 0.2 closes. Delta V1, paged sparse updates, and the synchronous presentation ledger are implemented, but the D milestone remains open until worker dependency groups and host frame-ticket effects integrate with that ledger and the complete D exit evidence is durable.

## Additive Delta V1

Snapshot-only consumers do not need to change. Delta-aware consumers initialize or resynchronize with a complete snapshot, then submit transactions against the exact accepted base:

```ts
const result = runtime.acceptDelta({
  schemaVersion: 'voxel.render-delta/1',
  worldId,
  epoch,
  baseRevision: acceptedRevision,
  revision: nextRevision,
  operations: [
    { op: 'put-resource', resource: updatedGeometry },
    { op: 'remove-chunk', key: oldChunkKey, incarnation: oldChunkIncarnation },
  ],
});

if (result.status === 'resync-required') {
  // Rebuild one complete snapshot from authoritative simulation state.
} else if (result.status === 'rejected') {
  throw new Error(`${result.code} at ${result.path}: ${result.message}`);
}
```

`resync-required` checks world, epoch, and exact base in that order without traversing stale operation payloads. Puts and removals are order-independent and references are checked against the complete final graph. Recreated keys must use an incarnation greater than the recorded per-lane tombstone.

Keyed instance patches cannot change geometry/material references, optional color/animation lane presence, or shadow participation. Each upsert supplies complete values for its keys:

```ts
runtime.acceptDelta({
  schemaVersion: 'voxel.render-delta/1',
  worldId,
  epoch,
  baseRevision,
  revision,
  operations: [{
    op: 'patch-batch-instances',
    key: batchKey,
    incarnation,
    revision: nextBatchRevision,
    removeInstanceKeys: removedKeys,
    upserts: { instanceKeys, matrices, colors },
  }],
});
```

Existing keys retain their external order, removed keys stable-compact, and new keys append in sorted order. The implementation stores instance lanes in fixed-size copy-on-write pages and gives the Three presenter exact changed ranges, so a sparse patch does not copy or upload the whole unchanged batch.
